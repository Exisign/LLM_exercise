LLM + Bedrock + RAG 프로젝트 최소 구성 리뷰

이 문서는 현재 프로젝트를 데이터 엔지니어 10년차 관점에서 본 정리다. 결론부터 말하면 지금 필요한 핵심은 Streamlit UI FastAPI API Bedrock LLM RAG 검색 네 가지이고, LangGraph 에이전트는 “반드시 지금 써야 하는 필수 구성”이라기보다 도구 호출, 반복 판단, 다중 단계가 실제 요구사항이 되었을 때 넣는 확장 축에 가깝다.

표기 규칙: 필요 지금은 보류 위험/수정 우선 같은 배경색은 서로 연결되는 내용이다. 글자색은 지정하지 않아 밤/낮 모드의 기본 글자색을 따른다.

1. 한 줄 결론

최소 제품 관점의 정답

“식사 추천 챗봇”을 안정적으로 만들려면 먼저 질문을 받고, 필요한 식당 데이터를 검색하고, 검색 결과를 프롬프트에 넣고, Bedrock 모델로 답변을 생성하는 2-step RAG면 충분하다. LangGraph는 이 흐름이 “모델이 스스로 검색 여부를 판단해야 한다”, “여러 도구를 골라 써야 한다”, “검증 후 재시도해야 한다”로 커질 때 도입하는 것이 맞다.

가장 먼저 고칠 부분

llm/__init__.py가 import될 때 AWS_BEARER_TOKEN_BEDROCK 값을 출력한다. 이건 기능 문제가 아니라 보안 문제다. 데모 코드라도 바로 제거해야 한다.

2. 공식 문서 기준으로 본 큰 흐름

LangChain 공식 문서는 RAG를 “외부 지식을 질의 시점에 가져와 LLM 답변에 보강하는 방식”으로 설명한다. 특히 LLM은 전체 사내 문서를 한 번에 넣을 수 없고, 학습 시점 이후 정보나 내부 데이터에 약하므로 검색 단계가 필요하다. 이 프로젝트의 “가게명, 메뉴, 특징, 가격” 데이터는 정확히 그 검색 대상이다.

1. 화면 app.py
Streamlit 채팅 입력과 세션 메시지 표시.
2. API server.py
POST /chat 요청을 받고 입력 구조를 검증.
3. 검색 rag_store.py
FAISS에서 질문과 가까운 식당 데이터를 검색.
4. 추론 llm
Bedrock 모델에 프롬프트와 검색 결과를 전달.
5. 응답 UI로 답변을 돌려주고 대화 이력에 저장.

화면 → API → 검색 → LLM → 화면

공식 문서에서 확인한 개념 현재 프로젝트에서의 의미 알아야 하는 이유
FastAPI는 Pydantic BaseModel로 request body 구조를 선언한다. UserRequest(query: str)는 프런트에서 넘어온 JSON을 검증하는 입구다. API 경계에서 데이터 모양을 고정해야 뒤쪽 LLM/RAG 코드가 덜 흔들린다.
Streamlit의 st.chat_input, st.chat_message, st.session_state는 채팅 UI와 세션 상태를 만든다. app.py는 얇게 유지하고, 추천 로직은 서버로 보낸다. 화면 코드는 자주 바뀐다. 추천 로직을 화면에 넣으면 테스트와 재사용이 어려워진다.
AWS Bedrock Converse API는 메시지 기반 대화 인터페이스와 시스템 프롬프트, 추론 설정, 도구 설정을 제공한다. ChatBedrockConverse를 한 곳에서 만들고 모든 체인이 공유하게 한다. 모델, 리전, 토큰, temperature가 파일마다 다르면 같은 질문도 다르게 동작한다.
LangChain Retrieval 문서는 2-step RAG를 “검색 후 생성” 구조로, Agentic RAG를 “모델이 검색 여부를 결정”하는 구조로 구분한다. 현재 식사 추천은 2-step RAG가 더 단순하고 빠르다. 에이전트는 유연하지만 지연 시간과 디버깅 비용이 늘어난다.
LangGraph는 State, Node, Edge로 워크플로우를 정의하고, MessagesState는 메시지 누적에 쓰인다. 현재 lg_rag_agent.py는 그래프 모양만 있고 실제 도구 실행 경로가 닫혀 있다. 도구 호출을 쓸 거라면 ToolNode와 tools_condition까지 연결해야 한다.

3. 현재 파일별 판단

파일 현재 역할 판단 정리 방향
app.py Streamlit 채팅 화면과 FastAPI 호출. 필요 응답 JSON 파싱과 assistant 오타를 고치고, API URL은 환경변수로 빼면 충분하다.
server.py /chat 백엔드 API. 필요 현재는 llm.chain만 호출한다. RAG를 쓸 거면 여기서 RAG 체인 또는 graph 하나만 호출하게 정리한다.
llm/__init__.py Bedrock LLM과 few-shot 프롬프트 구성. 수정 우선 환경변수 출력 제거. 특히 토큰 출력은 금지. LLM 생성 함수만 남기는 편이 안전하다.
langgraph/rag_store.py 더미 식당 데이터 임베딩, FAISS 검색. 필요 import 시점에 AWS 임베딩과 FAISS 구축이 실행된다. lazy loading 또는 빌드/서빙 분리로 바꿔야 한다.
langgraph/tools.py RAG 검색을 LangChain tool로 감싼다. 조건부 필요 LangGraph 에이전트를 쓸 때만 필요하다. 2-step RAG만 쓸 거면 tool wrapper 없이 함수 호출로 충분하다.
langgraph/lg_rag_agent.py LangGraph 기반 에이전트 실험. 지금은 과함 에이전트 학습용으로는 좋지만, 현재는 thinking → END라 도구가 실행되지 않는다.
rag/rag_test1.py, rag/rag_test2.py, rag/rag_test3.py RAG 학습/실험 스크립트. 운영 코드 아님 scripts/build_index.py와 rag_store.py로 역할을 나누고, 나머지는 예제 폴더로 둔다.
hp-story/ FAISS 저장 인덱스. 학습용 필요 저장된 FAISS를 로드할 때 pickle 역직렬화가 들어간다. 신뢰한 파일만 로드해야 한다.

4. 반드시 알아야 할 것과 지금 신경 덜 써도 되는 것

반드시 알아야 할 것
  • RAG는 재학습이 아니다. 검색 결과를 프롬프트에 붙여 답변 품질을 높이는 방식이다.
  • 임베딩은 검색용 숫자 변환이다. 사용자의 질문과 문서를 같은 벡터 공간에 놓고 가까운 항목을 찾는다.
  • 인덱싱과 질의는 분리한다. 문서 읽기, 청크 분할, 임베딩 저장은 배치 작업이고, 채팅 요청은 저장된 인덱스를 검색하는 런타임 작업이다.
  • API 경계는 구조화한다. FastAPI의 Pydantic 모델은 Java의 DTO처럼 생각하면 된다.
  • import 부작용을 줄인다. 모듈을 가져오는 순간 AWS 호출, 비밀 출력, 인덱스 생성이 일어나면 운영에서 예측이 어렵다.
지금은 신경 덜 써도 되는 것
  • MCP, A2A, 다중 에이전트: 현재 데이터와 기능 규모에서는 아직 빠르다.
  • 벡터 DB 제품 비교: 지금은 FAISS로 검색 품질과 흐름을 검증하는 것이 먼저다.
  • 파인튜닝: 식당 데이터 추천은 RAG 업데이트로 해결할 문제다.
  • 복잡한 LangGraph 상태 설계: 한 번 검색하고 답하면 되는 흐름에는 LCEL 체인이 더 읽기 쉽다.
  • 프롬프트 예시 과다: few-shot은 2~4개면 시작하기 충분하다. 데이터 정확도는 RAG가 담당해야 한다.

5. 현재 코드의 주요 문제

우선순위 문제 왜 문제인가 최소 수정 방향
1 llm/__init__.py에서 AWS 토큰 출력 로그, 콘솔, 캡처 화면으로 비밀값이 유출될 수 있다. print(os.getenv(...)) 제거. 필요한 경우 리전/모델 ID만 마스킹해서 logging.
2 app.py가 res.json() 전체 dict를 markdown에 넣음 서버는 {"response": "..."}를 반환하므로 화면에는 문자열만 꺼내야 한다. result = res.json().get("response", "")로 처리.
3 assistnat 오타 Streamlit chat role이 깨져 이전 대화 렌더링이 이상해질 수 있다. assistant로 수정.
4 lg_rag_agent.py가 llm.bind_tools()를 해도 ToolNode로 가지 않음 모델이 tool call을 만들 수는 있지만 실제 RAG 함수 실행 경로가 없다. LangGraph를 쓸 경우 ToolNode, tools_condition, tool 후 LLM 재호출 edge 연결.
5 rag_store.py가 import 시점에 임베딩과 FAISS를 생성 서버 시작이 느려지고, 테스트/import만 해도 AWS 비용과 실패 가능성이 생긴다. get_vector_db() 같은 lazy 함수로 감싸거나 저장 인덱스를 로드.
6 실험 코드와 실행 코드가 섞임 어떤 코드가 운영 경로인지 초보자가 구분하기 어렵다. scripts/, app/, examples/처럼 역할별로 분리.

6. 최소 구성으로 통합하기

현재 구조는 학습에는 좋지만, 운영 흐름으로 보면 LLM 생성 코드가 여러 파일에 흩어져 있다. 최소화하려면 “한 요청이 지나가는 길”을 하나로 만든다.

추천 구조: 먼저 2-step RAG

project/
  app.py                    # Streamlit: 화면만 담당
  server.py                 # FastAPI: /chat API만 담당
  meal_agent/
    config.py               # 환경변수 로드, 모델 ID, 리전
    llm.py                  # ChatBedrockConverse 생성
    prompts.py              # system prompt, few-shot
    rag_store.py            # FAISS load/search
    service.py              # answer(query): 검색 → 프롬프트 → LLM
  scripts/
    build_index.py          # 문서/식당 데이터 → 청크 → 임베딩 → FAISS 저장
  examples/
    rag_test1.py
    rag_test2.py
    rag_test3.py

이 구조에서 RAG는 데이터 검색 책임만 갖고, LLM은 답변 생성 책임만 갖는다. FastAPI는 두 기능을 호출하는 얇은 입구다. Java 관점으로 보면 service.py가 Service 계층, rag_store.py가 Repository 계층에 가깝다.

LangGraph를 꼭 유지할 때의 최소 연결

LangGraph 공식 문서 기준으로 도구 호출 패턴은 보통 LLM node → tools_condition → ToolNode → LLM node 형태다. 현재처럼 thinking → END만 있으면 tool call이 실행되지 않는다. 

from langgraph.graph import StateGraph, MessagesState, START
from langgraph.prebuilt import ToolNode, tools_condition

tools = [rag_search]
llm_with_tools = llm.bind_tools(tools)

def call_llm(state: MessagesState):
    return {"messages": [llm_with_tools.invoke(state["messages"])]}

builder = StateGraph(MessagesState)
builder.add_node("llm", call_llm)
builder.add_node("tools", ToolNode(tools))
builder.add_edge(START, "llm")
builder.add_conditional_edges("llm", tools_condition)
builder.add_edge("tools", "llm")
graph = builder.compile()
판단

위 코드는 “에이전트가 RAG 검색이 필요한지 판단하게 만드는” 최소 형태다. 하지만 지금 서비스가 항상 식당 데이터를 참고해야 한다면 에이전트보다 2-step RAG가 낫다. 검색이 항상 필요한데 모델에게 “검색할까 말까”를 묻는 것은 지연 시간과 실패 지점을 늘린다.

7. 파트별로 직관적으로 이해하기

Streamlit UI

Streamlit은 Python만으로 빠르게 화면을 만드는 도구다. st.chat_input은 사용자가 메시지를 입력하는 창이고, st.chat_message는 채팅 말풍선 컨테이너다. st.session_state는 브라우저 세션 동안 유지되는 dict라고 보면 된다.

Java 개발자 관점에서는 React 상태 관리보다 훨씬 단순하다. 다만 Streamlit은 입력이 들어올 때 스크립트가 다시 실행되는 모델이라, 대화 이력은 반드시 st.session_state.messages 같은 곳에 저장해야 한다.

FastAPI

FastAPI의 BaseModel은 Java의 request DTO처럼 생각하면 된다. UserRequest에 query: str라고 쓰면, 클라이언트가 보낸 JSON에 query가 있는지 확인하고 Python 객체처럼 다룰 수 있게 해준다.

지금 서버는 모든 예외를 {"response": "에러 ..."}로 반환한다. 데모에서는 편하지만, 운영에서는 HTTP status와 로그를 분리해야 장애 원인을 찾기 쉽다.

Bedrock LLM

ChatBedrockConverse는 Bedrock Converse API 기반의 LangChain chat model이다. 공식 문서상 Converse는 여러 Bedrock 모델에 대해 일관된 메시지 인터페이스를 제공하고, 시스템 프롬프트와 추론 파라미터를 다룰 수 있다.

여기서 중요한 것은 모델 호출 코드를 한 곳으로 모으는 것이다. llm/__init__.py, rag_test3.py, lg_rag_agent.py가 각자 모델을 만들면 모델 ID, temperature, token 설정이 갈라진다.

RAG와 FAISS

RAG는 “검색해서 가져온 근거를 LLM에게 같이 보여주는 방식”이다. FAISS는 벡터 유사도 검색 엔진이다. 사용자가 “가벼운 식사”라고 물으면, 임베딩 모델이 그 문장을 숫자 벡터로 바꾸고, FAISS가 비슷한 벡터를 가진 식당 설명을 찾아준다.

현재 rag_store.py는 더미 식당 데이터를 바로 임베딩해서 메모리에 올린다. 데이터가 5개일 때는 괜찮지만, 문서가 커지면 rag_test2.py처럼 인덱스를 한 번 만들어 저장하고 런타임에는 로드만 해야 한다.

청크와 임베딩

긴 문서는 한 번에 검색하지 않는다. RecursiveCharacterTextSplitter는 문단, 줄, 공백 순서로 최대한 자연스러운 단위를 유지하며 텍스트를 자른다. chunk_overlap은 경계에서 문맥이 끊기는 문제를 줄이려고 앞뒤 청크를 일부 겹치게 하는 값이다.

Java 관점으로 비유하면, 원문 전체를 한 row로 저장하는 대신 검색 가능한 작은 row 여러 개로 정규화하는 작업에 가깝다. 단, 너무 작게 자르면 문맥이 부족하고, 너무 크게 자르면 검색 정확도와 비용이 나빠진다.

LangGraph

LangGraph는 상태를 가진 워크플로우를 graph로 표현한다. State는 공유 메모리, Node는 함수, Edge는 다음에 실행할 노드 규칙이다. Java로 치면 상태 객체를 들고 여러 handler를 이동하는 workflow engine에 가깝다.

지금 프로젝트에서 LangGraph를 이해할 때 핵심은 하나다. 모델이 도구 호출을 결정해도, 그래프에 ToolNode가 연결되어 있지 않으면 도구는 실행되지 않는다. 그래서 현재 thinking_node만 있는 구조는 “에이전트처럼 보이지만 실제로는 단일 LLM 호출”이다.

8. 실무 기준 정리 순서

  1. 보안 llm/__init__.py의 환경변수 출력 제거.
  2. UI app.py에서 응답 dict 처리와 assistant 오타 수정.
  3. LLM Bedrock 모델 생성 코드를 한 모듈로 통합.
  4. RAG 더미 데이터 검색은 rag_store.py에 두되, import 시 즉시 생성하지 않게 변경.
  5. 인덱싱 문서 기반 RAG는 scripts/build_index.py로 분리.
  6. API server.py는 answer(query) 하나만 호출하게 단순화.
  7. Graph LangGraph는 학습 목적이면 examples로 이동하고, 운영에 쓸 때만 ToolNode까지 연결.

9. 코드 최소화 원칙

원칙 현재 프로젝트에 적용 초보자가 기억할 말
한 책임은 한 파일에 둔다. LLM 생성, 프롬프트, RAG 검색, API, UI를 섞지 않는다. “한 파일이 두 가지 이유로 바뀌면 쪼갤 때다.”
인덱싱과 서빙을 분리한다. rag_test2.py는 빌드 스크립트, rag_store.py는 검색 모듈로 나눈다. “검색할 때마다 DB를 새로 만들지 않는다.”
모델 설정은 중앙화한다. MODEL_ID, AWS_REGION, token 설정을 한 곳에서 읽는다. “같은 앱이면 같은 모델 설정을 써야 한다.”
그래프는 필요할 때만 쓴다. 항상 검색하는 메뉴 추천은 2-step RAG, 선택적 도구 호출은 LangGraph. “조건과 반복이 없으면 그래프가 아니라 함수 호출이면 된다.”
비밀값은 절대 출력하지 않는다. AWS 토큰 print 제거. “디버깅 로그도 유출 경로다.”

10. 공식 문서 근거와 내 판단 구분

내용 구분 근거
RAG는 외부 지식을 검색해 LLM 답변에 보강하고, 2-step RAG와 Agentic RAG는 지연 시간과 유연성에서 차이가 있다. 공식 문서 기반 LangChain Retrieval 문서
현재 프로젝트에는 2-step RAG가 먼저이고 LangGraph는 나중이어도 된다. 내 판단 현재 요구사항이 단일 검색 기반 메뉴 추천이고, 도구 선택/반복/승인 요구가 아직 없기 때문.
LangGraph는 State, Node, Edge를 사용하고 MessagesState는 메시지 누적에 적합하다. 공식 문서 기반 LangGraph Graph API 문서
lg_rag_agent.py는 현재 도구 실행이 되지 않는 반쪽짜리 graph다. 내 판단 코드상 ToolNode와 조건부 edge가 주석 처리되어 있고 thinking → END만 연결되어 있기 때문.
FAISS 저장 인덱스 로드 시 allow_dangerous_deserialization=True는 신뢰한 파일에만 써야 한다. 공식 문서 기반 LangChain FAISS API/통합 문서의 pickle 역직렬화 경고.
환경변수 출력은 즉시 제거해야 한다. 내 판단 공식 문서 특정 항목이라기보다 일반 보안 운영 원칙. 현재 코드에 실제 secret 출력이 있음.

11. 참고한 공식 문서

  • LangChain Retrieval - RAG 필요성, 2-step RAG와 Agentic RAG 비교.
  • LangGraph Graph API - State, Node, Edge, MessagesState 개념.
  • LangChain Tools - ToolNode와 tools_condition 라우팅 패턴.
  • LangChain ChatBedrock / ChatBedrockConverse - Bedrock chat model 연동과 tool calling 지원.
  • Amazon Bedrock Converse API - 메시지 기반 대화, 시스템 프롬프트, inferenceConfig, toolConfig.
  • LangChain FAISS Vector Store - FAISS 검색, retriever 변환, 저장/로드.
  • LangChain FAISS API Reference - load_local과 allow_dangerous_deserialization.
  • LangChain RecursiveCharacterTextSplitter - 청크 크기와 overlap 개념.
  • FastAPI Request Body - Pydantic BaseModel 기반 요청 구조.
  • Streamlit Chat Elements와 Session State - 채팅 입력, 메시지 컨테이너, 세션 상태.
